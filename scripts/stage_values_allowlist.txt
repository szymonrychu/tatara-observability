# Closed-set label VALUE allowlist for scripts/check_metric_provenance.py.
#
# metrics_allowlist.txt validates metric NAMES only. A rule can select a metric
# that IS emitted while filtering on a stageReason=/stage=/kind=/agent_kind= value
# that is NOT a member of the contract's closed set - the series never appears,
# and because every alert file sets default_no_data_state: "OK" the rule reports
# OK forever. Same failure class as a dead metric name, one level down (fix V7-7:
# this is exactly how a v6-era rule shipped a reference to the retired
# "pod-not-ready" stageReason and would have passed a name-only provenance check).
#
# Sections below are `## <label name>` followed by one value per line. Regenerate
# by re-reading the named contract section in
# docs/superpowers/plans/2026-07-12-task-centric-CROSS-REPO-CONTRACT.md and
# reconciling by hand; these are closed sets, not samples.

## kind:exempt-metrics
# `kind` is an OVERLOADED label name. On the Task family it is the A.4 origin kind
# (the closed set above); on the metrics below it means something else entirely, so
# the closed-set sweep must not be applied to them. The default is to CHECK: a new
# metric that overloads `kind` fails CI until someone lands it here deliberately.
# Each entry names what its `kind` label actually carries, verified in the operator
# worktree's internal/obs.
#
# scm_metrics.go: kind = the ACCESS CLASS of an SCM call ("write"|"read"), which is
# how dashboards/operator.json splits the write path out of the SCM counter.
operator_scm_writes_total
# operator_metrics.go: kind = the reconciled CR kind (Task|Project|Repository|...).
operator_reconcile_total
# contract K.1: kind = the CR kind whose etcd object is being sized / rejected /
# spilled / adopted, not the Task's origin kind.
operator_object_size_bytes
operator_object_too_large_total
operator_comment_spill_total
operator_orphan_adopted_total
